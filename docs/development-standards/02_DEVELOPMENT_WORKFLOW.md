# Development Workflow

版本：v0.1.0  
状态：生效  
最后更新：2026-04-26

## 核心原则

- 一次只推进一个明确目标。
- 每个任务必须能独立验证。
- 每次完成后更新 `DEVELOPMENT_TASKS.md`。
- 功能、文档、测试和资源预算必须同步推进。

## 标准流程

1. 阅读入口文档：`README.md`、`DEVELOPMENT_TASKS.md`、相关 `docs/` 专题文档。
2. 确认当前任务：明确输入、输出、验收方式和影响文件。
3. 先补测试或静态检查：任务涉及行为变化时，优先加入能失败的测试或 smoke 断言。
4. 小步实现：只改必要文件，不做无关重构。
5. 运行验证：默认至少运行 `npm test`；涉及 UI 时还要检查浏览器入口或 smoke 页面。
6. 更新文档：任务看板、README、CONTRIBUTING、CHANGELOG 和专题文档按影响范围同步。
7. 收尾报告：说明改动、验证结果、剩余风险。

## 模块插件方向的流程要求

- 新增格式前，先判断属于 `format-basic`、`format-plugin` 还是 `optional-plugin`。
- `format-plugin` 和 `optional-plugin` 默认不得进入核心包。
- 需要下载或加载插件时，必须有 manifest、资源预算、local-only 声明和失败降级策略。
- 插件能力上线前，必须证明核心包体积和默认依赖没有膨胀。

## 禁止行为

- 未更新 tasks 就结束开发。
- 未验证就声称完成。
- 将重依赖直接加入默认 dependencies。
- 为了某个格式绕过 `DocumentModel`，形成不可维护的私有直连。
- 默认上传用户文件、错误详情或转换结果。
